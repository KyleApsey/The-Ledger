## Vision

The Ledger is an internal bar prep PWA for Belle's Lounge (Ferndale, MI). This CEO plan covers the admin management interface added to the original Approach B prep tool design (approved 2026-06-12 in av7944-unknown-design-20260612-114148.md).

The admin interface solves the second half of the paper problem: staff have been replacing paper prep sheets, but managers still maintain recipes and pars on paper in a binder. This closes the loop — adding a new recipe, adjusting seasonal pars, or toggling an item off for the season takes 30 seconds instead of printing new sheets.

---

## Scope (V0 — integrated mobile-first admin tab)

All 5 cherry-pick expansions accepted:

- **D3.1 Recipe archiving:** archive + restore instead of hard delete. `archived_at` nullable timestamp. Staff-facing recipe view filters `WHERE archived_at IS NULL`. Archived recipes are restorable.
- **D3.2 Seasonal items:** `is_active` boolean on Items. Admin toggles items off-season items. Staff checklist filters `WHERE is_active = true`.
- **D3.3 Seasonal par presets:** new `ParPreset` table. Manager saves named par snapshots. "Activate" writes each item's `par_level` directly from the snapshot (in a transaction). Stale item references (deactivated items in the snapshot) are skipped; response includes `{ updated: N, skipped: M }`. Toast: "Pars updated — M items no longer exist and were skipped."
- **D3.4 Ingredient autocomplete:** `SELECT DISTINCT name FROM recipe_ingredients WHERE name ILIKE $prefix% LIMIT 20`, triggered after 2 characters.
- **D3.5 Recipe duplication:** deep copy (recipe + all ingredients + all steps) in one transaction, new UUID, name = "Copy of [Original]", `archived_at = null`. After duplication, navigate directly to the new draft in edit mode.

**NOT in V0 scope:** item reordering, bulk import, change log/audit trail (par level history deferred to V1).

## Architecture

### Admin tab placement
Same PWA as the prep tool. A "Manage" tab appears in the bottom nav only for `is_admin` staff. Same card/form UX language.

### Authorization model (two-layer)
- **Client layer:** `is_admin` from sessionStorage gates Manage tab visibility. Nuxt route middleware redirects non-admin staff from `/manage/*` to `/` with toast: "Admin access required."
- **Server layer:** Nuxt server middleware for `/api/admin/*` re-queries `SELECT is_admin FROM staff WHERE id = $staff_id` on every request. The `is_admin` value in sessionStorage is NEVER trusted server-side.
- **Staff management** (`/api/admin/staff`): protected by a separate env-var admin password (not staff PIN + is_admin).

### ParPreset activation
Direct write: `UPDATE items SET par_level = snapshot_value WHERE id = item_id` wrapped in a DB transaction. If the transaction fails, all updates roll back. Items not found in the DB (stale references) are skipped without error. Activation emits a structured log event.

### Recipe form rendering
Recipe instructions: stored as plain `TEXT` in `recipe_steps`, rendered via `{{ }}` interpolation in Vue templates. Never `v-html`. Prevents XSS from admin-entered content.

### Ingredient autocomplete
`GET /api/admin/recipes/ingredient-names?q=gin` → `SELECT DISTINCT name FROM recipe_ingredients WHERE name ILIKE $q% LIMIT 20`. Triggered client-side after 2 characters.

### Recipe validation (server-side)
- At least 1 ingredient required
- At least 1 step required
- `base_yield_liters` > 0 required
- `name` non-empty, max 200 chars
- Returns 400 with field-level errors on failure

## Data Model Changes

### New columns
```sql
ALTER TABLE items ADD COLUMN is_active BOOLEAN NOT NULL DEFAULT true;
ALTER TABLE recipes ADD COLUMN archived_at TIMESTAMP;
```

### New table
```sql
CREATE TABLE par_presets (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        TEXT NOT NULL,
  par_snapshot JSONB NOT NULL,   -- { item_id: par_level, ... }
  created_at  TIMESTAMP NOT NULL DEFAULT now(),
  activated_at TIMESTAMP         -- nullable; set on each activation
);
```

### New indexes
```sql
CREATE INDEX CONCURRENTLY items_is_active_idx ON items WHERE is_active = true;
CREATE INDEX CONCURRENTLY recipes_archived_at_idx ON recipes WHERE archived_at IS NULL;
CREATE INDEX recipe_ingredients_name_idx ON recipe_ingredients (name);
```

All base tables (`items`, `recipes`, `recipe_steps`, `recipe_ingredients`, `batches`, `waste_logs`, `staff`, `pars_check_sessions`) are created fresh in the initial migration — the ALTER statements above are for any migration strategy that starts from a seeded base.

## API Surface

### Staff routes (unchanged)
`GET /api/items` — now includes `WHERE is_active = true` filter.
Recipe fetch — now includes `WHERE archived_at IS NULL` filter.

### New admin routes (PIN session + is_admin re-verified server-side)
```
GET/POST   /api/admin/items
PATCH      /api/admin/items/:id
PATCH      /api/admin/items/:id/toggle       → flip is_active

GET/POST   /api/admin/recipes
PATCH      /api/admin/recipes/:id
POST       /api/admin/recipes/:id/archive    → SET archived_at = now()
POST       /api/admin/recipes/:id/restore    → SET archived_at = null
POST       /api/admin/recipes/:id/duplicate  → deep copy, navigate to new draft

GET/POST   /api/admin/presets
POST       /api/admin/presets/:id/activate   → UPDATE items in transaction

GET        /api/admin/recipes/ingredient-names?q=prefix
```

## UX Flows

### Admin: Items
1. Manage tab → Items list (active items first, inactive dimmed or in separate section)
2. "+ Add Item" FAB → Item Form
3. Tap existing item → Item Edit Form
4. Toggle → inline is_active flip with confirmation toast

### Admin: Recipes
1. Manage tab → Recipes list (active; "Show archived" toggle)
2. "+ Add Recipe" FAB → Recipe Form
3. Tap existing recipe → Recipe Edit Form
4. Three-dot menu:
   - Archive → confirm dialog → recipe removed from active list
   - Restore (if archived) → recipe returns to active list
   - Duplicate → deep copy → navigate directly to new "Copy of X" in edit mode
5. Duplicate button enters loading/disabled state on first tap; re-enables on server response

### Admin: Par Presets
1. Manage tab → Par Presets list
2. "+ Save Current Pars" → enters name → POST → appears in list (shows created_at)
3. Tap preset → "Activate these pars?" confirm → POST activate → toast: "Pars updated (N items, M skipped)"

### Non-admin redirect
Nuxt route middleware on `/manage/*`: if not `is_admin`, redirect to `/`, toast: "Admin access required."

## Observability

Admin mutations emit structured log: `{ action, entity_type, entity_id, performed_by: staff_id, timestamp }` via Nitro logger.

Actions logged: item created, item updated, item is_active toggled, recipe created, recipe updated, recipe archived, recipe restored, recipe duplicated, preset created, preset activated.

## Error handling

| Codepath | Failure | Response | User sees |
|---------|---------|----------|-----------|
| Admin item save | Constraint violation (missing container_volume) | 400 | "Container volume required for Cambro-measured items" |
| Admin recipe save | 0 ingredients | 400 | "At least one ingredient is required" |
| Admin recipe save | 0 steps | 400 | "At least one step is required" |
| Recipe archive | Recipe not found | 404 | Toast: "Recipe not found" |
| Recipe duplicate | Recipe not found | 404 | Toast: "Recipe not found" |
| Preset activate | Preset not found | 404 | Toast: "Preset not found" |
| Preset activate | Partial: stale item refs | 200 + skipped count | Toast: "Pars updated — N items skipped" |
| Preset activate | DB transaction failure | 500 | Toast: "Update failed — try again" |
| Any admin mutation | DB connection | 500 | Toast: "Something went wrong — try again" |

## Known V0 Limitations

- Waste logging does not update the stock snapshot (manual re-entry required)
- Last-write-wins for concurrent pars session edits (non-blocking toast on conflict)
- Par level history not tracked: activating a preset overwrites item par_levels without audit (V1)
- Admin changes to par_level / is_active while a checklist session is open: staff client sees stale data until refresh (V1: "par levels have changed — refresh" notification)
- No PIN lockout: accepted for internal trust model; revisit if deployment scope expands (P3 TODO)
- /admin/staff uses a separate env-var admin password; not integrated with is_admin PIN flow

## TODOS.md additions from this review

### TODO: PIN rate limiting (P3)
**What:** Add server-side failed-PIN rate limit: 5 attempts locks for 15 min per IP.
**Why:** Currently brute-forceable in minutes (10,000 combinations). Internal trust model is acceptable for now, but expansion to internet-facing or multi-location use would require it.
**Effort:** S
**Depends on:** None — can be added any time post-V0.

### TODO: Open session stale data notification (P3)
**What:** When admin activates a preset or toggles item is_active, show a non-blocking "Par levels have been updated — tap to refresh" toast to any staff with an open pars session.
**Why:** V0 limitation: admin changes don't propagate to open sessions. Staff see stale flagging logic until reload.
**Effort:** M (requires server-sent events or polling for admin change events)
**Depends on:** V0 shipped and in use — collect real feedback on whether this matters in practice.

1. Create all base tables: items, recipes, recipe_steps, recipe_ingredients, batches, waste_logs, staff, pars_check_sessions, par_presets
