## Supabase Client (Nitro server routes)

The `@supabase/supabase-js` client is initialized per-request inside each Nitro server route. No shared Nitro plugin state (Vercel serverless functions are isolated per invocation).

```ts
// Pattern used in every server route
import { createClient } from '@supabase/supabase-js'

export default defineEventHandler(async (event) => {
  const config = useRuntimeConfig()
  const supabase = createClient(config.supabaseUrl, config.supabaseSecretKey)
  // ...
})
```

Do NOT use `@nuxtjs/supabase` module — it is designed for Supabase Auth and is incompatible with the custom PIN auth used in this project.

## Authentication: JWT Session

PIN auth uses stateless JWTs signed with `SESSION_SECRET`. Library: `jose` (works in Node.js and edge runtimes without polyfills).

### Flow

```
POST /api/auth/login { pin: string }
→ SELECT id, name, is_admin, pin_hash FROM staff
→ bcryptjs.compare(pin, pin_hash)
→ sign JWT { sub: staff_id, is_admin: bool, exp: now + 7200 }
→ return { token, staff_name, is_admin }
```

Client stores token in `sessionStorage`. Sent as `Authorization: Bearer <token>` on every subsequent request.

### Session expiry

JWT `exp` = issue time + 2 hours. The client also tracks last interaction time (`mousedown`, `touchstart`, `keydown`). If client-side idle > 2hr, clear sessionStorage and redirect to PIN screen. The 2hr inactivity timeout is enforced client-side; the JWT provides the server-side expiry bound.

No `/api/auth/refresh` route — if the token expires, the user re-enters their PIN.

### PIN storage

PINs stored as bcrypt hashes in the `staff` table (`bcryptjs` npm package). Plaintext PIN is never persisted.

```sql
-- staff table
pin_hash TEXT NOT NULL  -- bcrypt hash of the 4-digit PIN
```

### Server-side verification

Nitro middleware for `/api/*` routes:
1. Extract Bearer token from `Authorization` header
2. Verify JWT signature and expiry using `jose`
3. Attach `{ staff_id, is_admin }` to event context

Nitro middleware for `/api/admin/*` routes (additional step):
4. Re-query `SELECT is_admin FROM staff WHERE id = $staff_id` — the JWT `is_admin` is used for logging only; the DB value is authoritative for admin authorization

## PostgreSQL Functions (Required for RPC calls)

`@supabase/supabase-js` uses PostgREST's HTTP API, which cannot express DISTINCT queries, multi-row transactions, or ON CONFLICT DO NOTHING natively via the query builder. Three PostgreSQL functions are required and must be included in `002_functions.sql`.

### 1. activate_par_preset

```sql
CREATE OR REPLACE FUNCTION activate_par_preset(p_preset_id UUID)
RETURNS JSON
LANGUAGE plpgsql
AS $$
DECLARE
  v_snapshot JSONB;
  v_item_id UUID;
  v_par_level NUMERIC;
  v_updated INT := 0;
  v_skipped INT := 0;
BEGIN
  SELECT par_snapshot INTO v_snapshot
  FROM par_presets
  WHERE id = p_preset_id;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'preset_not_found';
  END IF;

  FOR v_item_id, v_par_level IN
    SELECT key::UUID, value::NUMERIC
    FROM jsonb_each_text(v_snapshot)
  LOOP
    UPDATE items SET par_level = v_par_level WHERE id = v_item_id;
    IF FOUND THEN
      v_updated := v_updated + 1;
    ELSE
      v_skipped := v_skipped + 1;
    END IF;
  END LOOP;

  UPDATE par_presets SET activated_at = now() WHERE id = p_preset_id;

  RETURN json_build_object('updated', v_updated, 'skipped', v_skipped);
END;
$$;
```

Called via: `supabase.rpc('activate_par_preset', { p_preset_id: presetId })`

### 2. duplicate_recipe

```sql
CREATE OR REPLACE FUNCTION duplicate_recipe(p_recipe_id UUID)
RETURNS UUID
LANGUAGE plpgsql
AS $$
DECLARE
  v_new_id UUID := gen_random_uuid();
BEGIN
  INSERT INTO recipes (id, name, base_yield_liters, archived_at, created_at)
  SELECT v_new_id,
         'Copy of ' || name,
         base_yield_liters,
         NULL,
         now()
  FROM recipes
  WHERE id = p_recipe_id;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'recipe_not_found';
  END IF;

  INSERT INTO recipe_ingredients (id, recipe_id, name, quantity, unit, sort_order)
  SELECT gen_random_uuid(), v_new_id, name, quantity, unit, sort_order
  FROM recipe_ingredients
  WHERE recipe_id = p_recipe_id;

  INSERT INTO recipe_steps (id, recipe_id, step_order, instruction)
  SELECT gen_random_uuid(), v_new_id, step_order, instruction
  FROM recipe_steps
  WHERE recipe_id = p_recipe_id;

  RETURN v_new_id;
END;
$$;
```

Called via: `supabase.rpc('duplicate_recipe', { p_recipe_id: recipeId })`

### 3. get_ingredient_suggestions

```sql
CREATE OR REPLACE FUNCTION get_ingredient_suggestions(q TEXT)
RETURNS TABLE(name TEXT)
LANGUAGE sql
AS $$
  SELECT DISTINCT ri.name
  FROM recipe_ingredients ri
  WHERE ri.name ILIKE q || '%'
  ORDER BY ri.name
  LIMIT 20;
$$;
```

Called via: `supabase.rpc('get_ingredient_suggestions', { q: prefix })`

## Deployment (CI/CD)

Vercel GitHub integration handles deployment. No GitHub Actions required.

1. Connect Vercel project to the `the_ledger` GitHub repo
2. Set `main` as the production branch
3. Set environment variables in Vercel Dashboard (SUPABASE_URL, SUPABASE_SECRET_KEY, SESSION_SECRET)
4. Every push to `main` → Vercel builds + deploys automatically
5. Pull request preview deployments use the same env vars (configure in Vercel project settings)

## Row Level Security (RLS)

RLS is **disabled** for V0. All server routes use the service role key, which bypasses RLS. Application-level auth in Nitro middleware (JWT verification + `is_admin` re-query) is the single control point.

RLS can be added later as a defense-in-depth measure if the project moves to internet-facing deployment with multiple access patterns.
